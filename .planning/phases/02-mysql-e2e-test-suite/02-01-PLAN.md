<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| test process → querylex binary | Subprocess invoked via os/exec; inherits env vars set by t.Setenv (HOME, QUERYLEX_*). No host env leakage. |
| querylex binary → MySQL container | MySQL connection via DSN with root:testpass — well-known Docker test credentials, not production secrets. Docker random port limits network exposure. |
| test process → filesystem | Temp dir created by t.TempDir(); automatically cleaned up. Encrypted credential file uses AES-256-GCM with e2e-test-passphrase. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | test/mysql/setup_test.go (hardcoded testpass) | accept | Well-known Docker test password (root:testpass); no production secrets; Docker container is ephemeral tmpfs |
| T-02-02 | Elevation of Privilege | compose.yaml MySQL service (ports: ["0:3306"]) | accept | Docker assigns random host port; container is ephemeral (tmpfs); CI-only; no persistent data |
| T-02-03 | Information Disclosure | RunQuerylex subprocess env inheritance | mitigate | t.Setenv(HOME, t.TempDir()) ensures binary reads synthetic workspace, not user's real ~/.querylex; QUERYLEX_* vars set to test-only values |
| T-02-04 | Tampering | EncryptedFileStore passphrase via env var | accept | QUERYLEX_KEYCHAIN_PASSPHRASE is a test-only convenience; in production, passphrase is prompted interactively; test passphrase is well-known "e2e-test-passphrase" |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No new external packages installed in this phase (all Go stdlib or already in go.mod) |
</threat_model>
